## 2. 核心问题：什么是"记忆"？

### 多数人的误区

大多数AI项目怎么做？
> 把所有对话、所有会话都存下来 → 称之为"记忆"

**这只是日志文件，不是真正的记忆！**

### 真正的记忆应该包含四类信息

| 类型 | 定义 | 例子 | 存储方案 |
|------|------|------|----------|
| **身份** | 你是谁 | 名字、职业、基础信息 | SQL |
| **习惯** | 你做什么 | 每天早上做什么、健身频率 | Vector DB |
| **偏好** | 你喜欢什么 | 口味、办公习惯 | Vector DB |
| **事件/任务** | 你要做什么 | 日历、待办事项 | SQL |

> "其他都是噪音。大多数你对AI说的话不需要被存储。"

## 3. 技术架构演进

### 阶段1：混合存储方案

**问题**：自然语言查询无法直接查SQL
- 你不能说"查一下我告诉你关于健身计划的内容"
- 你需要知道确切的表名和字段

**解决方案**：混合存储

```
┌─────────────────────────────────────────┐
│           用户自然语言查询              │
└─────────────────┬───────────────────────┘
                  ↓
        ┌────────┴────────┐
        ↓                 ↓
    SQL数据库        Vector数据库
   (Postgres)        (Pinecone)
        ↓                 ↓
   精确查询          语义搜索
        └────────┬────────┘
                 ↓
          合并结果
```

- **SQL (Postgres/Supabase)**：存结构化数据（身份、任务）
- **Vector DB (Pinecone)**：存需要语义检索的数据（习惯、偏好）

---

### 阶段2：让LLM直接读写 - 失败

**最初方案**：
```
给LLM访问两个数据库的工具 → 让它自己决定什么时候读写
```

**结果：灾难！**

- LLM会产生幻觉，会写错记忆分类
- 检索到无关信息
- **最致命**：当系统的核心是"可靠记忆"时，幻觉是致命的

### 阶段3：节点化架构 - 成功

```
用户输入
    ↓
┌────────────────────────────────────┐
│ [分割节点]                          │
│ 输入 → 分离出：                     │
│   - memory_to_write（要写入的）     │
│   - memory_to_fetch（要读取的）     │
│   - ignore（忽略的）                │
└────────────────────────────────────┘
    ↓
┌────────────────────────────────────┐
│ [分类节点]                          │
│ 给每部分打标签：                     │
│   identity / habit / preference /   │
│   event / task                     │
└────────────────────────────────────┘
    ↓
┌────────────────────────────────────┐
│ [路由节点]                          │
│   ├──→ [记忆写入] → 并行写          │
│   │            Pinecone + Postgres  │
│   └──→ [记忆读取] → 并行读          │
│                获取相关上下文        │
└────────────────────────────────────┘
    ↓
┌────────────────────────────────────┐
│ [最终回答节点]                       │
│ 聚合上下文 → 单次LLM调用 → 回答     │
└────────────────────────────────────┘
```

---

### 两个关键决策

#### 1. 并行读写

- **问题**：顺序执行延迟太高
- **解决**：并行读写，响应时间大幅下降

#### 2. LLM只在必要时使用

| 任务 | 推荐方案 |
|------|----------|
| 分割输入 | 规则/正则 |
| 分类记忆 | 规则/正则 |
| 读写记忆 | 数据库操作 |
| **生成回答** | **LLM（必须的）** |

> "结果是一个端到端可预测的系统。如果出错，我知道哪个节点失败了。这比一个黑盒幻觉好一百万倍。"

## 5. 核心教训

### 教训1：设计先行

> **"存什么"比"怎么存"更重要**

- 大多数人直接跳进技术选型
- 应该先回答设计问题：你的AI需要记住什么？
- 很多信息其实不需要存储

### 教训2：延迟是真实问题

- 记忆系统的延迟是真实存在的挑战
- 并行检索和写入是必须的，不是可选项
- 用户不会等待一个慢悠悠的AI助手

### 教训3：LLM不是万能的

- LLM很贵（token费用）
- LLM不可预测（幻觉问题）
- 尽可能用确定性逻辑替代LLM调用

### 教训4：调整范围不是失败

作者最初想做硬件设备（桌面机器人），后来决定先专注软件层：

> "硬件是倍增器。如果记忆架构有问题，物理设备只会让问题更糟。如果记忆架构稳固，硬件就是包装问题——不是根本问题。"

## 6. 对开发者的启示

### 如果你想做类似项目

1. **先想清楚存什么**
   - 不是所有对话都需要记住
   - 明确你的AI需要"认识"用户哪些方面

2. **用合适的技术存合适的数据**
   - 结构化信息（任务、日程）→ SQL
   - 模糊匹配（习惯、偏好）→ 向量数据库

3. **不要让LLM做所有事情**
   - 能用规则就不用AI
   - LLM很贵且不可预测

4. **并行处理**
   - 读写可以同时做
   - 减少用户等待时间

5. **可预测性 > 智能**
   - 一个能解释自己错误的系统 > 一个看起来聪明但会幻觉的系统
